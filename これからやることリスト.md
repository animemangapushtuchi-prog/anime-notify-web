# Animiru これからやることリスト（レイアウト完成 → 広告ユニット設置の前段）

広告ユニットは「レイアウトの完成形＝コンテンツの厚み」ができてから。
この段では【集客/SEO】と【中核価値＝作品情報の充実】を軸に進める。

---

## ① 詳細画面：放送（地上波・BS・CS）と 配信（サブスク）を分離

**現状**：配信サービス（AniList由来・日本関連のみ）を1つの欄で表示。テレビ局の放送時刻は「次の1回」しか使っていない。

**やること**
- 欄を2分割する。
  - 「テレビ放送」欄：局ごとに、初回放送日時＋リピート放送を表示。
  - 「配信（サブスク）」欄：Netflix / dアニメ / Prime 等をブランドアイコン＋作品ページ直リンク。
- テレビ放送の表示イメージ（局ごとにまとめる）：
  - AT-X：2026/7/8〜 毎週水 23:30（リピート：毎週金 11:30 ／ 毎週火 17:30）
  - TOKYO MX：2026/7/8〜 毎週水 25:00
  - BS日テレ：2026/7/9〜 毎週木 25:00

**データ源と課題**
- しょぼいカレンダー（`cache/tvSchedule`＝programs: title / stTime / chName …）に局名と放送時刻がある。今は「次の1件」だけ使用。
- 追加処理：作品ごとに **全放送を集計 → 局ごとにグループ化**（初回＋リピートを判定）。
- 注意：現在のキャッシュは短期間ぶんの可能性。局×初回＋リピートまで綺麗に出すには、しょぼいカレンダーの取得範囲拡大 or 番組詳細の利用（バックエンド拡張）が要るかも。まず「既存データで出せるぶん」から段階的に。
- ついでに回収：過去の積み残し「配信ブランドアイコン」もこの欄で対応。

## ② おすすめ / まとめページ（特設ランキング）← 集客の主砲

**狙い**：「おすすめアニメ○選」系はSEOで強い。しかも上位に個人勢が食い込む余地あり（レッドオーシャンに見えて穴がある）。Next.js（SSR/SSG）は“canvas描画のFlutter”と違い、この土俵で戦える＝乗り換えた意味がここで効く。

**ナビ追加**：下タブに「おすすめ」を追加（ホーム / 検索 / **おすすめ** / 通知）。

**私が作る「型」（テンプレート）**
- ルート：`/osusume`（特設一覧）と `/osusume/[slug]`（各特設ページ）。
- 特設ページ構成：
  1. メインビジュアル（ヒーロー画像）＋タイトル（例：「2026夏 異世界アニメ おすすめ20選」）＋導入文
  2. ランキング形式（1位…N位）。各エントリ：
     - 順位バッジ／作品名／メインビジュアル画像
     - **レビュータイトル＋レビュー文**（← あなたが用意：Gemini草稿→確認）
     - **配信リンク**（Amazon Prime / Netflix / dアニメ 等）＝将来アフィリエイト化の器
     - **CTA**「この作品を登録して新着通知を受け取る」＝アプリ登録・使用への自然な導線
  3. 下部に総括＋関連特設への内部リンク（回遊で滞在・SEO強化）
- テーマ例：今期配信 / 異世界 / スポーツ / ジャンプ原作 / 泣ける / 100話超え … 何本でも量産可。

**データの持ち方（要相談：AかB）**
- 案A（推奨・大量向き）：リポジトリに **MDX/JSON** で記事を置く → SSGで最速・SEO最強・Git管理。
  - 分業：あなたがGemini草稿 → 私が取り込み＆整形。1日10〜20本、特設5本規模でも回しやすい。
- 案B：Firestoreに記事コレクション＋アプリ内の簡易管理フォーム（ブラウザだけで編集したいなら）。

**SEO仕込み（型に含める）**
- 各ページに `metadata`（title / description / OGP画像）、構造化データ（ItemList）、`sitemap.xml`、パンくず、内部リンク。

**収益**
- まとめページは **アフィリエイト**（Amazonアソシエイト等）と特に相性◎。AdSenseと二本柱に。

## ③ 開発者用ダッシュボード（リアルタイム可視化）

**狙い**：利用者数・アクセス数・登録作品の傾向を、デバッグ／運用のため一望。

**やること**
- `/admin/dashboard`（あなたのUID or 合言葉でガード。一般非公開）。
- 指標例：
  - 総ユーザー数／アクティブ数
  - 登録作品ランキング（どの作品が人気か）＝今後の特設ネタにも使える
  - 通知送信数、端末プラットフォーム比率
  - アクセス数：Vercel Analytics（or GA）連携
- 実装：Firestore集計をリアルタイム購読。件数が増えたら全件読みは重いので、**集計ドキュメント（`cache/stats`）をCloud Functionsで定期更新**する形が安全。
- ※「開くたびに最新値を出すライブ画面」はこのアプリの得意分野。まず簡易版→拡充。

## ④ 検索：ページ送り／無限スクロール

**現状**：検索は1ページぶんのみ。

**やること**
- AniList GraphQL は page / perPage でページング可能。**IntersectionObserver で下端到達時に次ページを追記**（無限スクロール）。
- 併せて「検索候補サジェスト（入力中のドロップダウン）」も候補（過去の積み残し）。

---

## 積み残し（過去セッション分の小項目）

- 声優・スタッフ行タップ → その人物名で検索
- パスワードリセット（メール無し設計のため、再発行方式を要検討）
- ProfileMenuの「準備中」：プロフィール編集、アプリ版案内（お問い合わせは mailto 化済み）
- 通知文の作り込み（「『見たい』作品に新情報！」等）＝バックエンドのプッシュ文面
- 「大きな情報」イベント（adapt＝続編・アニメ化）の生成 → 「見た/中断」ステータスへの通知を実体化（今は器だけ）
- AdSense：EU向け同意メッセージ（GDPR）設定 ＊審査はブロックしない

---

## 収益・運用（②③と並行で仕込む）

- アフィリエイト（Amazonアソシエイト等）を配信リンク／まとめページに。
- AdSense：審査通過後に広告ユニット設置（レイアウト完成後）。同意メッセージも設定。

---

## 推奨する着手順（たたき台）

1. **④ 検索の無限スクロール** … 軽い・すぐ効くUX改善。ウォームアップに最適。
2. **① 詳細の放送/配信分離** … 中核価値。まず既存データで出せるぶんから。
3. **② おすすめ特設の「型」** … 集客の主砲。器さえ作ればあなたが量産開始できる。
4. **③ 開発者ダッシュボード** … 運用が回り出したら。

＊② の「型」を早めに作っておくと、あなたの記事量産（Gemini草稿→確認）と並走できて時間効率が良い。
